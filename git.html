After installation, this doc can be displayed with git Myhelp

CLONE / FETCH / CHECKOUT

  • git clone <filename>.git <directory> # filename can be an url (git@XXX.git, https://XXX.git) or a file (local or distant host:XXX.git)
  • git fetch <remote> # to get branches from a remote without merging
  • git pull <remote> # to get and merge from a remote
  • git checkout <branch>
  • git checkout <branch> <file> # checkout a file
  • git remote -v # list the configured remotes
  • git rename <remote_old> <remote_new>
  • git remote add <remote> <url>

LOG / STATUS

  • git status
  • git log
  • git log1 # equivalent to git log -n1
  • git log --graph --oneline –all
  • gitk (graphical interface)
  • git show <HASH> with -s to get commit message, --name-only to list modified files, --name-status to list modified files with their status
  • git show <branch>:<file> # use s instead of show to open it in vi
  • git diff <branch1>:<file1> <branch2>:<file2> # use d to show diff in vi

BRANCHES

  • git branch # list branches (with -r for remote)
  • git checkout # to go on a branch
  • git checkout -b <new branch name> <branch> # create a new branch starting from other branch
  • git branch -d <branch> # to remove local branch
  • git push <remote> -d <branch> # to remove remote branch (never use it)
  • git branch <branch>; git reset --keep HEAD~2 # move the two last commits on a new branch

COMMIT / PUSH

  • git add <file or directory>
  • git commit
  • git commit --amend # to modify last commit
  • git reset <HASH> --hard # reset local branch to this commit (remove commits)
  • git push <remote> +<HASH>:<branch> # reset remote branch to this commit (remove commits, never use it)
  • git pull; git push # push a commit/branch (the pull is done to check if local repository is up to date)

TAGS

A good practice would be to push only “annotated tags” (created with the -a option) and keep “lightweight tags” for local development.

  • git tag -a <TAG> # add a tag to the current commit
  • git tag -a <TAG> <HASH> # add a tag to a given commit
  • git push <TAG> # to push a tag
  • git fetch --tags # to get tags
  • git lasttag # to find the last tag
  • git tag -d <TAG> # suppress a local tag
  • git push --delete <remote>upstream <TAG> # suppress a remote tag

MERGE

  • git merge --no-commit --no-ff <branch>; git merge --abort dry run
  • git merge <branch> # merge a branch in the current one
  • in case of a conflict:
    • git merge --abort # to abort
    • git status # to get the list of files where there is a conflict
    • edition of files, conflicts are identified by ‘<<<<<’, ‘>>>>>’ and ‘=====’ strings
    • git mergetool to use vi (in the order: version before merge, last common version, version to merge, result)
  • git cherry-pick <HASH> # merge a single commit

REBASE

  • git checkout <branch>; git rebase main # to rebase branch on main
  • git checkout <branch_dev>; git rebase --onto <branch_A> <branch_B> # if branch_dev is based on branch_B, put commits between B and dev onto A (eg. backphasing)

STASHES

Staches is a way to save uncommited changes to be able to perform other action (eg. merge)

  • git stash save "message" # to create a stash, add the -u option to include non tracked files
  • git stash list # to get a list of the stashes
  • git stash show stash@{n} # n being the stash number, to inspect a stach. With -p to see a diff
  • git stash pop stash@{n} # apply the stash number n
  • git stash branch <branch> stash@{n} # to put the stash number n in a branch
  • git stash drop stash@{n} # to delete the stash number n

GITHUB PULL REQUEST

The pull request ID can be found using the web interface

  • git fetch <remote> pull/<ID>/head:<branch> # create a local branch with the pull request content
  • git pull <remote> pull/<ID>/head # merge the pull request in the current branch
